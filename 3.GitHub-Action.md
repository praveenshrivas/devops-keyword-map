## 1. Core Concepts

| Keyword | What it is |
|---|---|
| **Workflow** | YAML file in `.github/workflows/` |
| **Event** | Trigger for a workflow (`on:`) |
| **Job** | Set of steps that run on a runner |
| **Step** | Individual task within a job (action or shell command) |
| **Action** | Reusable unit of code (Docker, JavaScript, or composite) |
| **Runner** | VM/container executing jobs (GitHub-hosted or self-hosted) |
| **Workflow run** | Single execution instance of a workflow |

---

## 2. Triggers / Events

- `push`, `pull_request`, **`pull_request_target`** ⚠️
- `workflow_dispatch` — manual trigger, with inputs
- `workflow_call` — reusable workflows
- `schedule` — cron syntax
- `repository_dispatch` — external/webhook trigger
- `release`, `issues`, `issue_comment`, `fork`, `create`, `delete`
- `workflow_run` — trigger off another workflow's completion
- Event filters: `paths`, `paths-ignore`, `branches-ignore`
- Activity types: `types: [opened, closed, synchronize]`

<details>
<summary>💡 <b>Gotcha: pull_request vs pull_request_target</b></summary>

- `pull_request` — runs with **read-only** token, no access to secrets, uses the PR's merge commit. Safe default for fork PRs.
- `pull_request_target` — runs in the **context of the base repo**, has access to secrets, checks out base branch by default. Using it carelessly with fork PR code = classic supply-chain risk.

</details>

## 3. Workflow Syntax

```yaml
name: CI
on: [push]
permissions:
  contents: read

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install
        working-directory: ./app
```

- `runs-on` — `ubuntu-latest`, `windows-latest`, `macos-latest`, self-hosted labels
- `uses` vs `run`
- `with` — action inputs
- `env` — workflow/job/step level
- `id` — step identifier for referencing outputs
- `if` conditionals — `success()`, `failure()`, `always()`, `cancelled()`
- `continue-on-error`
- `timeout-minutes`
- `shell` — bash, pwsh, python, cmd

---

## 4. Jobs

- `needs` — job dependency graph
- **`strategy.matrix`** — matrix builds (multi-OS/version testing)
- `max-parallel` / `fail-fast` — matrix controls
- `outputs` — job-to-job data passing
- `environment` — deployment environments, protection rules
- `concurrency` — cancel-in-progress, concurrency groups
- `container` — run job inside a Docker container
- `services` — sidecar containers (Postgres, Redis for tests)

---

## 5. Actions

| Type | Notes |
|---|---|
| **Docker container action** | Runs inside a container |
| **JavaScript action** | Runs directly on the runner (Node.js) |
| **Composite action** | Bundles multiple steps into one reusable action |

- Marketplace staples: `actions/checkout`, `actions/setup-node`, `actions/cache`, `actions/upload-artifact`, `actions/download-artifact`
- `action.yml` — metadata file defining an action
- **Versioning actions** ⚠️ — `@v4` vs `@sha` vs `@main` (pin to SHA for supply-chain safety)

---

## 6. Secrets & Variables

- Repository secrets / Organization secrets / Environment secrets
- `secrets` context (`${{ secrets.NAME }}`)
- `vars` context — repo/org/environment-level **non-sensitive** variables
- **`GITHUB_TOKEN`** — auto-generated token, scope it via `permissions:`
- Encrypted secrets — sealed via public key, never exposed in logs
- **OIDC (OpenID Connect)** — federated auth to AWS/Azure/GCP, no static cloud credentials

---

## 7. Contexts & Expressions

```yaml
${{ github.sha }}
${{ github.ref }}
${{ needs.build.outputs.artifact_name }}
${{ contains(github.event.head_commit.message, 'skip-ci') }}
```

- `github`, `env`, `job`, `steps`, `runner`, `matrix`, `needs` contexts
- Functions: `contains()`, `startsWith()`, `endsWith()`, `format()`, `join()`, `toJSON()`, `fromJSON()`, `hashFiles()`
- Status functions: `success()`, `failure()`, `cancelled()`, `always()`

- Called workflow `inputs` / `outputs` / `secrets: inherit`
- Starter workflow templates

## 11. Security

- **`permissions:` block** ⚠️ — least-privilege `GITHUB_TOKEN` (default to `contents: read`)
- CODEOWNERS integration
- **Dependabot** — dependency updates, security alerts
- **CodeQL** — code scanning
- Secret scanning / push protection
- **OIDC federated identity** — cloud role assumption without static keys
- Third-party action pinning to SHA — supply chain security
- Branch protection rules tied to required status checks

## 13. Monitoring / Debugging

- Job summary — `$GITHUB_STEP_SUMMARY`
- Status badges

## 15. Advanced Patterns

- Monorepo path filtering — only trigger on changed paths
- Conditional job execution based on branch/tag
- **Dynamic matrix generation** — via `fromJSON` on a prior job's output
- Release automation — tag-triggered builds, semantic-release
- Multi-cloud/multi-environment deploy pipelines
- Approval gates via required reviewers on environments

---

## 16. Comparison Points

> Coming from a Jenkins / GitLab CI mindset:

| | GitHub Actions | Jenkins |
|---|---|---|
| Config | YAML-native, in-repo | Groovy (Jenkinsfile) or UI |
| Architecture | Ephemeral, pull-based runners | Master/agent, long-lived |
| Extensibility | Marketplace actions | Plugin ecosystem |
| Reusability | Reusable workflows / composite actions | Shared libraries |
