## 2. Dockerfile Instructions

```dockerfile
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=build /app/dist ./dist
USER node
EXPOSE 3000
HEALTHCHECK CMD wget -qO- http://localhost:3000/health || exit 1
CMD ["node", "dist/index.js"]
```

| Instruction | Notes |
|---|---|
| `FROM` | Base image, multi-stage `AS` |
| `RUN` | Build-time command execution |
| **`CMD` vs `ENTRYPOINT`** вљ пёЏ | Default command vs fixed executable |
| **`COPY` vs `ADD`** вљ пёЏ | `ADD` supports URLs/tar auto-extract; `COPY` is preferred/explicit |
| `WORKDIR` | Set working directory |
| **`ENV` vs `ARG`** вљ пёЏ | Runtime vs build-time variables |
| `EXPOSE` | Documents container port (doesn't publish it) |
| `VOLUME` | Declares a mount point |
| `USER` | Non-root best practice |
| `HEALTHCHECK` | Container-level health status |
| `.dockerignore` | Exclude files from build context |

<details>
<summary>рџ’Ў <b>Gotcha: CMD vs ENTRYPOINT</b></summary>

- `CMD` вЂ” default arguments, easily overridden by `docker run <image> <other-command>`.
- `ENTRYPOINT` вЂ” fixed executable; `CMD` becomes its default arguments if both are present.
- Combined form `ENTRYPOINT ["executable"]` + `CMD ["default-args"]` is the standard pattern for CLI-style images.

</details>

## 3. Image Building

```bash
docker build -t myapp:latest -f Dockerfile .
docker buildx build --platform linux/amd64,linux/arm64 -t myapp:latest .
```

- Build context
- **Layer caching** вЂ” each instruction = a layer, cache invalidation rules
- **Multi-stage builds** вЂ” reducing final image size в­ђ
- **BuildKit** (`DOCKER_BUILDKIT=1`) вЂ” improved caching/parallelism
- `docker buildx` вЂ” multi-platform builds
- Union filesystem (OverlayFS)
- Squashing layers
- Distroless / minimal base images (`alpine`, `scratch`, distroless)
- Image digest (`sha256`) vs tag

## 4. Container Lifecycle

```bash
docker run -d --name web -p 8080:80 -v data:/var/lib/data -e ENV=prod --restart unless-stopped nginx

docker stats
```

- `docker start / stop / restart / pause / unpause`
- `docker attach`
- `docker top`
- `docker rm` / `docker rmi`
- `docker ps -a`
- **Restart policies**: `no`, `on-failure`, `always`, `unless-stopped`
- Container states: created, running, paused, exited, dead

## 5. Networking

| Network type | Use case |
|---|---|
| **bridge** (default) | Single-host container-to-container |
| **host** | Container shares host's network namespace |
| **none** | Fully isolated |
| **overlay** | Multi-host (Swarm) |
| **macvlan** | Container gets its own MAC/IP on the physical network |

- `docker network create/ls/inspect/connect/disconnect`
- Port mapping (`-p hostPort:containerPort`)
- **Custom bridge networks vs default bridge** вљ пёЏ вЂ” custom bridges get automatic DNS resolution by container name; the default bridge does not

## 7. Docker Compose

```yaml
services:
  web:
    build: .
    ports: ["8080:80"]
    depends_on: [db]
    environment:
      - ENV=prod
  db:
    image: postgres:16
    volumes: ["db-data:/var/lib/postgresql/data"]

volumes:
  db-data:
```

- `build` vs `image` in a service
- `depends_on`
- `docker compose up/down/build/logs/ps`
- Compose profiles
- Multiple compose files (`-f base.yml -f override.yml`)
- **Compose V2** (integrated `docker compose`) vs legacy `docker-compose` binary

## 10. Security

- User namespaces
- Non-root `USER` in Dockerfile
- **Docker socket exposure risk** вљ пёЏ вЂ” mounting `/var/run/docker.sock` = root-equivalent access to host

- Seccomp profiles / AppArmor / SELinux integration
- Read-only containers (`--read-only`)
- Resource limits (`--memory`, `--cpus`, `--pids-limit`)
- Least privilege base images
